# 섹션 31. 하네스(Harness) 엔지니어링 - 개념

> **이 섹션을 마치면**: Claude Code 하네스의 전체 구조를 이해하고, 왜 하네스 엔지니어링이 중요한지 설명할 수 있습니다.

---

## 하네스(Harness)란?

### 단어의 의미

```
Harness (하네스) = 마구(馬具)

말을 제어하기 위한 장비 전체를 의미합니다.
고삐, 안장, 등자를 연결하는 체계.

Claude Code에서 Harness = AI를 제어하고 자동화하는 전체 시스템
```

### Claude Code Harness 정의

```
Claude Code Harness =
  설정 파일(settings.json) +
  훅(Hooks) 시스템 +
  스킬(Skills) +
  메모리(CLAUDE.md) +
  MCP 서버 +
  권한(Permissions)

이 모든 것이 유기적으로 연결된 자동화 인프라
```

---

## 왜 하네스 엔지니어링인가?

### Before: 단순 사용

```
[사용자] → "요청" → [Claude Code] → "응답"

매번 같은 설명 반복
일관성 없는 결과
수동으로 확인 필요
```

### After: 하네스 엔지니어링

```
[설정된 하네스]
  ↓ (세션 시작 시 자동 컨텍스트 주입)
[사용자] → "요청" → [Claude Code]
  ↑ (자동 품질 검사)     ↓ (자동 실행)
[검증 결과]           [훅 자동 트리거]
  ↑                     ↓
[알림]           [포맷팅/테스트 자동화]

결과: 일관되고 안전하며 자동화된 AI 작업 환경
```

## 하네스의 구성 요소

```
┌─────────────────────────────────────────────┐
│              Claude Code Harness             │
│                                              │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │ CLAUDE.md │  │settings. │  │  Skills  │  │
│  │ (메모리)  │  │  json    │  │ (명령어) │  │
│  └──────────┘  └──────────┘  └──────────┘  │
│                                              │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │  Hooks   │  │   MCP    │  │Permission│  │
│  │ (이벤트) │  │ (연결)   │  │  (권한)  │  │
│  └──────────┘  └──────────┘  └──────────┘  │
│                                              │
└─────────────────────────────────────────────┘
```

### 각 구성 요소의 역할

| 구성 요소 | 역할 | 비유 |
|---------|------|------|
| CLAUDE.md | 프로젝트 컨텍스트 제공 | 직원 온보딩 문서 |
| settings.json | 동작 방식 제어 | 사규/규정집 |
| Skills | 반복 작업 자동화 | 업무 매뉴얼 |
| Hooks | 이벤트 기반 자동화 | 자동화 규칙 |
| MCP | 외부 시스템 연결 | 시스템 통합 |
| Permissions | 접근 권한 관리 | 보안 정책 |

## 하네스 엔지니어링의 3원칙

### 원칙 1: 일관성

```
목표: 어떤 개발자가 사용해도 같은 결과

방법:
- 팀 공용 CLAUDE.md (프로젝트 규칙)
- 공유 스킬 라이브러리
- 표준화된 훅 설정
```

### 원칙 2: 안전성

```
목표: 실수로 인한 피해 최소화

방법:
- 위험한 명령어 차단 (deny list)
- 중요 파일 보호 (.env, 설정 파일)
- 변경 전 자동 체크포인트
- 테스트 자동 실행
```

### 원칙 3: 자동화

```
목표: 반복 작업 제거

방법:
- PostToolUse 훅으로 자동 포맷팅
- Stop 훅으로 자동 테스트
- 스킬로 표준 작업 자동화
- MCP로 외부 작업 자동화
```

---

## 하네스 성숙도 모델

```
레벨 0 (없음):
  Claude Code 기본 사용, 설정 없음

레벨 1 (기초):
  CLAUDE.md + 기본 permissions 설정

레벨 2 (중급):
  스킬 라이브러리 + 기본 훅 설정 + MCP 연결

레벨 3 (고급):
  완전 자동화 파이프라인 + 팀 공유 설정 + 커스텀 에이전트

레벨 4 (마스터):
  프로젝트 특화 하네스 + 지속적 개선 시스템
```

## 하네스 vs 일반 사용의 차이

### 실제 예시

**하네스 없이:**
```
1. Claude에게 "코드 작성해줘" 요청
2. 코드 작성됨
3. 직접 prettier 실행 (기억해야 함)
4. 직접 eslint 실행 (기억해야 함)
5. 직접 테스트 실행 (기억해야 함)
6. 직접 git add/commit (기억해야 함)

→ 매번 수동, 잊으면 놓침
```

**하네스 있이:**
```
1. Claude에게 "코드 작성해줘" 요청
2. 코드 작성됨
3. [자동] prettier 실행
4. [자동] eslint 실행
5. [자동] 테스트 실행
6. [자동] git commit

→ 완전 자동, 절대 안 놓침
```

## 핵심 정리

```
하네스 엔지니어링 = Claude Code를 조직화하고 자동화하는 기술

구성 요소:
1. CLAUDE.md    → 컨텍스트 (무엇을 알아야 하나)
2. settings.json → 동작 방식 (어떻게 작동할까)
3. Skills       → 반복 명령어 (무엇을 자주 하나)
4. Hooks        → 자동 트리거 (언제 자동으로 실행할까)
5. MCP          → 외부 연결 (무엇과 연동할까)
6. Permissions  → 안전 장치 (무엇을 막을까)

목표: 일관성 + 안전성 + 자동화
```
